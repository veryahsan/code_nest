# Code Nest Application Flow

This guide explains what happens when you run the Code Nest Rails server. It is written for studying the application flow from the command line to the browser response.

Current project status: **Phase 0 Bootstrap**. The app has the Rails foundation, Hotwire/Tailwind UI shell, Sidekiq/Redis configuration, RSpec, CI, and deployment config. Authentication, organisations, teams, employees, projects, and Google Docs integration will be added in later phases.

---

## 1. The Big Picture

When you run:

```bash
bin/dev
```

Rails starts three processes through `Procfile.dev`:

```procfile
web: bin/rails server -p ${PORT:-3000}
css: bin/rails tailwindcss:watch
worker: bundle exec sidekiq -C config/sidekiq.yml
```

That means:

1. `web` starts the Rails web server using Puma.
2. `css` watches Tailwind files and rebuilds CSS when styles change.
3. `worker` starts Sidekiq for background jobs.

So the development app is not only "a server". It is a small local system:

Prefer `bin/dev` for local development. If you run `foreman start` directly,
Foreman reads `Procfile` instead of `Procfile.dev`. In this project both work,
but `bin/dev` is the intended command because it also starts the Tailwind watcher.

```text
Browser
  |
  v
Puma / Rails Web Process
  |
  +--> PostgreSQL database
  |
  +--> Redis cache / ActionCable / Sidekiq queue
  |
  +--> Sidekiq worker process
  |
  +--> Tailwind CSS watcher
```

## 2. What Happens When You Run `bin/dev`

### Step 1: `bin/dev` Runs Foreman

`bin/dev` is generated by Rails/Tailwind. It uses Foreman to read `Procfile.dev`.

Foreman starts each line in `Procfile.dev` as a separate local process:

- Rails web server
- Tailwind watcher
- Sidekiq worker

This lets you run the complete development stack with one command.

### Step 2: Rails Server Starts

The Rails process runs:

```bash
bin/rails server -p 3000
```

Internally this boots:

1. `config/boot.rb`
2. Bundler and all gems from `Gemfile`
3. Rails framework components from `config/application.rb`
4. Environment-specific config, for example `config/environments/development.rb`
5. Routes from `config/routes.rb`
6. Initializers from `config/initializers/*`
7. Puma web server from `config/puma.rb`

### Step 3: Tailwind Watcher Starts

Tailwind watches:

```text
app/assets/tailwind/application.css
app/views/**/*.erb
app/helpers/**/*.rb
app/javascript/**/*.js
```

When it detects utility classes like `bg-white`, `text-slate-900`, or `max-w-7xl`, it builds the final CSS into:

```text
app/assets/builds/
```

The Rails layout loads this CSS through:

```erb
<%= stylesheet_link_tag :app, "data-turbo-track": "reload" %>
```

### Step 4: Sidekiq Starts

Sidekiq starts with:

```bash
bundle exec sidekiq -C config/sidekiq.yml
```

It connects to Redis using `REDIS_URL` and waits for background jobs.

Current queues:

```yaml
critical
default
mailers
integrations
low
```

In Phase 0 there are no real jobs yet, but this is ready for later work like:

- sending emails
- syncing Google Docs
- processing webhooks
- sending notifications

## 3. Rails Boot Flow

The main boot file is:

```text
config/application.rb
```

It loads Rails components:

```ruby
require "active_record/railtie"
require "action_controller/railtie"
require "action_mailer/railtie"
require "action_cable/engine"
```

It also defines important application-level decisions:

```ruby
config.active_job.queue_adapter = :sidekiq
config.time_zone = ENV.fetch("APP_TIME_ZONE", "UTC")
```

And it autoloads architectural folders:

```text
app/services
app/facades
app/queries
app/decorators
app/serializers
app/forms
```

These folders match the architecture standards:

- Controllers stay thin.
- Controllers call facades.
- Facades coordinate complete user flows.
- Services perform focused business logic.
- Policies handle authorization.
- Serializers serve the JSON API.

## 4. Request Flow When You Open `localhost:3000`

When you visit:

```text
http://localhost:3000
```

the flow is:

```text
Browser
  |
  v
Puma receives HTTP GET /
  |
  v
Rails router checks config/routes.rb
  |
  v
root "home#show"
  |
  v
HomeController#show
  |
  v
app/views/home/show.html.erb
  |
  v
app/views/layouts/application.html.erb wraps the page
  |
  v
Browser receives HTML + CSS + JavaScript
```

### Route

The root route is defined in:

```ruby
root "home#show"
```

That means:

- controller: `HomeController`
- action: `show`
- view: `app/views/home/show.html.erb`

### Controller

Current controller:

```ruby
class HomeController < ApplicationController
  def show
  end
end
```

The action is empty because Rails automatically renders the matching view:

```text
app/views/home/show.html.erb
```

### Layout

The view is wrapped by:

```text
app/views/layouts/application.html.erb
```

That layout renders:

```erb
<%= render "shared/navbar" %>
<%= render "shared/flash" %>
<%= yield %>
<%= render "shared/footer" %>
```

The `yield` is where the current page view is inserted.

## 5. Frontend Flow

The app uses:

- Hotwire Turbo
- Stimulus
- TailwindCSS
- Importmap

### JavaScript Entry Point

Rails loads JavaScript from:

```text
app/javascript/application.js
```

Current imports:

```javascript
import "@hotwired/turbo-rails"
import "controllers"
```

This means:

- Turbo is enabled globally.
- Stimulus controllers are auto-loaded.

### Stimulus Controllers

Stimulus controllers live in:

```text
app/javascript/controllers/
```

Current custom controller:

```text
app/javascript/controllers/flash_controller.js
```

It automatically hides flash messages after a short delay.

The flash partial uses it like this:

```erb
data-controller="flash"
data-flash-delay-value="5000"
```

So when a flash message appears, Stimulus connects to that HTML element and removes it after 5 seconds.

## 7. Redis Flow

Redis is used for three things:

1. Sidekiq job queue
2. ActionCable real-time messages
3. Rails cache store

The shared connection is controlled by:

```text
REDIS_URL
```

Code Nest intentionally uses the default pure Ruby Redis client path. We avoid
the optional `hiredis-client` native driver because local Redis/Sidekiq blocking
reads can produce noisy `hiredis_read` timeout warnings on some macOS/Ruby
combinations.

Sidekiq configuration:

```text
config/initializers/sidekiq.rb
config/sidekiq.yml
```

The Sidekiq initializer also sets Redis socket timeouts. The read timeout is
longer than the default so Sidekiq can safely block while waiting for jobs
without logging noisy "Redis is offline" messages when the queue is empty.

ActionCable configuration:

```text
config/cable.yml
```

Production cache store:

```ruby
config.cache_store = :redis_cache_store
```

## 8. Background Job Flow

When application code later enqueues a job:

```ruby
SomeJob.perform_later(...)
```

the flow will be:

```text
Rails app
  |
  v
ActiveJob
  |
  v
Sidekiq adapter
  |
  v
Redis queue
  |
  v
Sidekiq worker process
  |
  v
Job performs work
```

This matters for features like Google Docs integration. For example:

```text
Google webhook received
  |
  v
Rails controller creates integration job
  |
  v
Sidekiq processes job async
  |
  v
DocumentationFacade syncs metadata
  |
  v
ProjectDocument updates in Postgres
  |
  v
Turbo Stream notification can be broadcast
```

## 9. Authorization Flow

The app uses Pundit.

Pundit is included in:

```text
app/controllers/application_controller.rb
```

Current behavior:

```ruby
include Pundit::Authorization
rescue_from Pundit::NotAuthorizedError, with: :user_not_authorized
```

Base policy:

```text
app/policies/application_policy.rb
```

By default, every action returns `false`:

```ruby
def show?
  false
end
```

This is intentional. It follows the security rule:

> Deny by default, then explicitly allow.

In Phase 1, once authentication and tenancy exist, policies will know:

- who the current employee is
- which organisation they belong to
- which team/project/document they can access

## 10. Facade and Service Flow

The architecture docs say controllers should call facades, not individual services.

Current base classes:

```text
app/facades/application_facade.rb
app/services/application_service.rb
```

Future example:

```text
ProjectsController#create
  |
  v
ProjectCreationFacade.call(...)
  |
  +--> CreateProjectService.call(...)
  +--> AssignEmployeesService.call(...)
  +--> SetupRemoteResourceService.call(...)
  +--> AttachGoogleDocService.call(...)
  |
  v
returns Result(success?, value, error)
```

This keeps controllers clean:

```ruby
result = ProjectCreationFacade.call(params)

if result.success?
  redirect_to result.value
else
  render :new, status: :unprocessable_entity
end
```

The app currently has the base structure ready, but the real domain facades will be added feature by feature.

## 12. Testing Flow

RSpec is configured through:

```text
.rspec
spec/spec_helper.rb
spec/rails_helper.rb
```

When you run:

```bash
bundle exec rspec
```

RSpec:

1. Loads `rails_helper`.
2. Boots the Rails test environment.
3. Checks pending migrations.
4. Loads support files from `spec/support`.
5. Runs specs.

Current smoke spec:

```text
spec/requests/home_spec.rb
```

It verifies:

- `GET /` returns HTTP 200
- response body includes `Code Nest`

Support files:

```text
spec/support/database_cleaner.rb
spec/support/factory_bot.rb
spec/support/sidekiq.rb
spec/support/devise.rb
```

`spec/support/sidekiq.rb` sets:

```ruby
Sidekiq::Testing.fake!
```

That means jobs are not executed automatically during tests. They are stored in fake queues so tests can inspect them.

```text
render.yaml
```

It defines:

- web service: Puma Rails app


Start script:

```text
bin/render-start.sh
```

It runs:

```bash
bundle exec puma -C config/puma.rb
```

Production request flow:

```text
User browser
  |
  v

  |
  v
Puma
  |
  v
Rails app
  |
  +--> Render Postgres
  +--> Render Redis / Key Value
```

## 15. Current Phase 0 Request Flow Summary

Right now the main browser flow is:

```text
bin/dev
  |
  +--> Puma Rails server
  +--> Tailwind watcher
  +--> Sidekiq worker

Browser visits /
  |
  v
config/routes.rb
  |
  v
HomeController#show
  |
  v
app/views/home/show.html.erb
  |
  v
app/views/layouts/application.html.erb
  |
  +--> shared/_navbar.html.erb
  +--> shared/_flash.html.erb
  +--> page content via yield
  +--> shared/_footer.html.erb
  |
  v
HTML response back to browser
```

As we build later phases, the same flow will grow to include:

- Devise authentication
- current employee lookup
- organisation/team scoping
- Pundit authorization
- facade calls
- service objects
- database writes
- Sidekiq jobs
- Turbo Stream broadcasts
- API serializers

The important idea is:

> Rails receives the request, routes it to a controller, the controller delegates business work to a facade, the facade coordinates services/models, Pundit protects access, and Rails renders either HTML for Hotwire or JSON for the API.
